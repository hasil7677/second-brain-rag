The Lineage

Where this idea came from

Four ideas, each building on the last. The final step is just replacing the human with an agent.

01 / CONCEPT
🧠
Tiago Forte — Building a Second Brain
The idea: offload your thinking into an external system so your brain is free to make connections, not memorise facts. Capture → Organise → Distill → Express. The system holds the knowledge; you hold the judgment.
concept
02 / IMPLEMENTATION
💎
Obsidian — Markdown + Wikilinks + Graph
Obsidian turned the second brain concept into a format: plain .md files with [[wikilinks]] between them, rendered as a force-directed knowledge graph. Human-readable, git-friendly, no proprietary lock-in.
format
03 / INSIGHT
⚡
Karpathy — "The LLM is the programmer"
Andrej Karpathy: "Obsidian is the IDE, the LLM is the programmer, the wiki is the codebase." The same format that humans use to think can be the context window an agent reasons over. Consumer changes, format stays the same.
insight
04 / FORMALISED
🏗️
Google OKF — A spec, not a library
Google formalised the format: Markdown + YAML frontmatter + [[wikilinks]] + a type: field. Backed by Dataplex for enterprise-scale cataloguing, lineage, and freshness. We implement the same format from scratch.
spec
What we built: Identical to Obsidian's internals — .md files, YAML frontmatter, [[wikilinks]], type fields. The only difference: the consumer is a RAG agent, not a human reading in a GUI. Same format. Different brain.
The Format

What an OKF file looks like

A knowledge file is just a Markdown document with structured YAML at the top. That's the entire spec.

claims_handling_sop--loss-adjustor-threshold.md OKF · type: claims_procedure
--- title: Loss Adjustor Appointment Threshold type: claims_procedure source: claims_handling_sop tags: [claims, loss-adjustor, threshold, field-assessment] links: - [[underwriting_guidelines--property-valuation-methods]] - [[claims_handling_sop--total-loss-declaration]] --- # Loss Adjustor Appointment Threshold A loss adjustor must be appointed for any claim exceeding $50,000 or where liability is disputed. For claims above $150,000, a senior loss adjustor is required. Field assessment is mandatory when structural damage is reported or when the insured disputes the initial desktop valuation. ## See also - [[underwriting_guidelines--property-valuation-methods]] - [[claims_handling_sop--total-loss-declaration]]
That's it. No database. No schema migrations. No API to call. A file you can read in any text editor, diff in git, and open in Obsidian. The agent reads the same file a human would.
The Pipeline

5 stages from question to answer

Every query passes through the same five stages. The interesting parts are stages 2, 3, and 4.

① STAGE
📥
INGEST
Parse .md files, extract YAML frontmatter, build in-memory corpus. Wikilinks become edges.
›
② STAGE
🔀
ROUTE
SLM (Mistral-7B) classifies the query: IN_CORPUS for known concepts, HYBRID for complex questions.
›
③ STAGE
🔍
RETRIEVE
BM25 keyword search + BGE-M3 semantic embeddings, fused with Reciprocal Rank Fusion (RRF).
›
④ STAGE
🔗
ENRICH
Follow [[wikilinks]] from top results up to 2 hops. Agent sees a web of knowledge, not isolated chunks.
›
⑤ STAGE
💬
GENERATE
Claude Haiku reads enriched context, generates a grounded answer. Sources attached.
The Knowledge Graph

What the corpus looks like as a graph

Each .md file is a node. Each [[wikilink]] is an edge. Cross-document links reveal hidden connections between policy, claims, and underwriting.

KNOWLEDGE GRAPH — 4 source documents · ~24 concept nodes · wikilink + cross-doc edges
Why this matters: When the agent retrieves "flood-zone-1-decline" from underwriting guidelines, it also follows the cross-doc edge to "flood-zone-classification" in the risk framework — without you having to chunk or join those documents manually. The wikilinks are the join logic.
Query Walkthrough

"Can we write flood cover for a Zone 2 property?"

Tracing a real query through all 5 pipeline stages.

①
ROUTE — SLM CLASSIFIER
Mistral-7B reads the query against all 24 concept titles. Detects flood-zone and underwriting — known corpus terms. Routes to IN_CORPUS path.
②
RETRIEVE — BM25 + BGE-M3 + RRF
BM25 scores on keyword overlap. BGE-M3 encodes the question and all 24 concepts to embeddings, takes cosine similarity. RRF fuses both ranked lists into a single score.
flood-zone-1-decline · rrf 0.82 flood-zone-classification · rrf 0.74 bfe-requirements · rrf 0.61
③
ENRICH — WIKILINK FOLLOW (2 hops)
flood-zone-1-decline links to [[property-valuation-methods]] and [[bfe-requirements]]. Both pulled into context. Agent now sees why Zone 1 is declined, not just that it is.
🔗 property-valuation-methods ← flood-zone-1-decline 🔗 bfe-requirements ← flood-zone-classification
④
GENERATE — CLAUDE HAIKU
Claude receives 5 enriched concept bodies as context. Generates: "Zone 2 properties are eligible for flood cover subject to a base flood elevation survey. Zone 1 is an automatic declination. Zone 2 requires a minimum sum insured of $350,000 and a flood-specific excess."
5 sources · 2 enriched ~1.8s total
Under the Hood

How hybrid retrieval actually works

Two retrieval signals fused into one. Neither alone is enough.

Signal How it works Good at Bad at
BM25 keyword Term frequency × inverse document frequency. Exact word match, no embeddings. Precise terms: "Zone 2", "co-insurance", "loss adjustor" Paraphrases, synonyms, semantic meaning
BGE-M3 semantic Encode query + all concepts to 768-dim vectors. Cosine similarity in embedding space. Meaning: "property damage" ≈ "structural loss" Rare terms, policy numbers, specific clause names
RRF fusion score = 1/(rank_bm25 + 60) + 1/(rank_semantic + 60). Rank-based, not score-based. Robust to score magnitude differences between signals Cannot distinguish when one signal is clearly superior
Why re-encode every query? Our corpus is 24 files. We re-encode all concepts fresh on every query (no persistent vector index). At 24 files this is fine — BGE-M3 takes ~80ms for the whole corpus. At 10,000 files you'd switch to Pinecone or Weaviate and persist the index.
